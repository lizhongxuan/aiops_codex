# UI 优化修改任务清单 (todo_ui_0324.md)

基于 `design_ui_0324.md` 的设计规范，将整体 UI 优化拆解为可执行的开发任务。建议按照以下 4 个阶段逐步实施：

## Phase 1: 卡片归一化体系建设
> 目标：建立 12 种核心卡片体系，清理冗余输出，理清不同消息卡片的边界。

- [ ] 1.1 前端新增卡片分发与渲染器逻辑，支持 12 种核心卡片（`UserMessageCard`, `AssistantMessageCard`, `ThinkingCard`, `PlanCard`, `CommandCard`, `FileChangeCard`, `CommandApprovalCard`, `FileChangeApprovalCard`, `ChoiceCard`, `ResultSummaryCard`, `NoticeCard`, `ErrorCard`）的类型映射。
- [ ] 1.2 清理冗余后端输出（禁止直接在聊天卡片正文透传 `status: completed`、`Turn completed` 及生硬技术性尾卡）。
- [ ] 1.3 拆分并重构现有的 `StepCard` 和 `ResultCard`，将单纯的结构化结论归入 `ResultSummaryCard`，技术状态过程归入过程摘要或对应业务步骤卡。
- [ ] 1.4 新增轻量级 `NoticeCard` 承接系统级提示（如重连成功、线程重建、主机切换等无关痛痒的弱提示）。
- [ ] 1.5 新增结构化 `ErrorCard`（需支持操作建议如 Retry/Refresh 动作，替代生硬的底层 RPC 报错卡）。
- [ ] 1.6 新增 `ChoiceCard` 支持通用选择输入（如 `tool/requestUserInput` 请求环境下拉等单选场景）。

## Phase 2: 对齐核心交互与视觉样式
> 目标：调整布局基线与宽度参数、优化字体呼吸感、并对特定对话组件与审批卡进行形式重构，全面提升对话沉浸感与企业级交互质感。

- [ ] 2.1 全局对话视窗与排版基线调整
  - [ ] 放大对话主内容列最大宽度到 `900px ~ 1024px` 级别，以保障代码块与表单展示体验。
  - [ ] 全面对话字号/行高规范化（正文提升至 `15~16px`、行高 `1.7` 左右提供宽裕间距）。
  - [ ] 优化元信息及时间线标注字号（降级至单记录 `12px` 左右，选用低对比度浅灰如 `text-gray-400`）。
- [ ] 2.2 用户气泡 (`UserMessageCard`) 重构
  - [ ] 取消气泡边框及尖角修饰，采用全包裹的 `rounded-2xl`（约16px）大圆角。
  - [ ] 调整气泡背景色为干净的浅灰色（如 `bg-gray-100` 或 `#F3F4F6`），搭配宽裕充沛的内外边距（如 `px-5 py-3`）。
- [ ] 2.3 命令审批卡 (`CommandApprovalCard`) 重构为结构化表单视图
  - [ ] 引入明确的意图题头文案与灰色背景包围的等宽命令预览区。
  - [ ] 将默认的同意/拒绝按钮升级为 Radio/List 单选项交互结构（例如：同意一次 / 同意且后续带相同前缀面询 / 拒绝再做调整）。
  - [ ] 底部补充前置运行环境（如本地容器）与权限作用域（如默认权限）的下拉带轻量图标选项组。
  - [ ] 调整底端操作栏：右侧强化纯黑底色的“提交并回车”按钮与轻巧的“跳过”操作。整体边框使用轻量阴影配合微线边框 (`border-gray-200`) 并加上大圆角轮廓。
- [ ] 2.4 命令回显态 (`CommandCard`) 后置时间线退化改造
  - [ ] 引入命令阶段特有的执行态展现（等宽深色背景内的流式输出临时终端）。
  - [ ] **执行完成退化动效**：成功运行结束后，将庞大的代码块或边框卡片隐褪，退化收敛至一条不干扰画面的横条时间线状态记录（如左描述“已运行 xxx 命令”，右附分割线与“已处理 38s”）。
  - [ ] 若命令执行失败，则取消折叠退化逻辑，保留完整报错输出区以供回查。
- [ ] 2.5 统一处理 `FileChangeApprovalCard` 与 `FileChangeCard`，同步沿用全新的圆角边距及行高选项设定。

## Phase 3: “正在思考”状态机与防假死反馈
> 目标：建立回合状态树，确保发信期间系统处于每秒可视感知态。

- [ ] 3.1 补充底层级渲染状态机制
  - [ ] Web 端引入基于 turn 运行时的细粒度状态树记录 (`idle`, `thinking`, `planning`, `waiting_approval`, `executing`, `finalizing`, `completed`, `failed`)。
- [ ] 3.2 植入 `ThinkingCard`
  - [ ] 用户消息敲入发出 150ms 左右即刻原位回显 Spinner 或打字机提示灯（如 `正在思考 / 正在等待确认`）。
  - [ ] 在 `CommandCard` 前置工作项拉起、或收到正文回馈流增量时，无缝移位或收起并淡出 `ThinkingCard`。
- [ ] 3.3 构建高级过程动作摘要条（顶部回显机制）
  - [ ] 对接系统事件，实现在头部原位单条实时状态更新，替代不停翻滚的啰嗦新卡片（例如实时渲染：`已浏览 2 个文件，1 个搜索，1 个列表`）。
  - [ ] 引入单文件检索动作提示动效（如搜索时第二行展示 `现在搜索网页（query）`，完毕后去重汇总进数字标签内；查阅单个文件时展示 `现在浏览 xxx`，完毕后合拢进列表中）。
- [ ] 3.4 追加带内嵌设计的 `Plan` Checklist 工作栏
  - [ ] 清理普通 `PlanCard` 突兀的展板感。
  - [ ] 将当前工作列改造为顺时序阅读的 CheckList（圆圈、步骤号），左上置入 `plan` 记号，右侧提供列表细项折叠/展开能力。

## Phase 4: `codex app-server` 断线稳定性与全屏终端页打通
> 目标：覆盖弱网恢复痛点，提供高级查验与人手接管操作台。

- [ ] 4.1 网络 Socket 稳定性及重连恢复可视化
  - [ ] 拦截并收敛 WebSocket 的异常断线和心跳异常。
  - [ ] 原位最多叠加重试更新 5 次的展现层提示（`Reconnecting... (1/5)` 逐级递增）。
  - [ ] 重试耗尽失败后进入 `stopped` 锁定态：输入框禁用、内容保留，在异常卡中提供显示的 `Retry / Refresh` 操作兜底按钮。
- [ ] 4.2 开发独立的高可用终端管理页面
  - [ ] 新建单页全屏路由映射向 `/terminal/:hostId` 节点。
  - [ ] 构建“顶底单主终端视窗 + 右侧可选信息抽屉面板”布局，上侧安置主机在线情况及重置操控栏（Ctrl+C/清屏 等）。
  - [ ] 基于 `xterm.js` 及 WebSocket PTY 会话流完成终端内容同步开发，联调 PTY 信号传导（Resize / 输入终止）。
- [ ] 4.3 会话与终端入口交互打通
  - [ ] 全局顶部 Header / 主机筛选器面板清单内打通按钮动作，一键实现“进入终端（目标机）”。
  - [ ] 执行卡（`CommandCard`）内提供悬浮“在终端中打开”按键，提供原机追溯上下文操作。
