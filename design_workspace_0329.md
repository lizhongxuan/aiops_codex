# 协作工作台体验优化设计 — 2026-03-29

## 一、信息架构精简

### 现状问题

当前 6 个页面：单机对话、主机管理、经验包库、协议工作台、终端、Agent Profile。左侧导航入口分散，运维工程师需要在多个页面之间跳转才能完成一个完整的工作流。

### 设计目标

把导航精简为三个核心入口，覆盖运维工程师的两个核心场景：
- "我要跟某台机器聊"（单机排障）
- "我要对一批机器做事"（批量运维）

### 具体方案

左侧导航改为：

```
对话          ← 合并当前 ChatPage + TerminalPage
工作台        ← 当前 ProtocolPage（协作工作台）
设置          ← 收纳主机管理、经验包库、Agent Profile
```

#### 对话页合并终端

- 对话页底部加一个可拖拽的终端面板（类似 VS Code 集成终端）
- 默认收起，点击"终端"按钮或快捷键展开
- 终端面板和聊天流同时可见，不需要跳页面
- 终端自动绑定当前选中的主机

#### 设置页收纳

- 主机管理、经验包库、Agent Profile 合并到一个设置页
- 用 tab 或左侧子导航切换
- 这些页面使用频率低，不需要占一级导航

#### 导航项的二级信息

每个导航项显示关键状态：

```
对话
  web-08 · 执行中

工作台
  mission: nginx 巡检 · 5/8 完成

设置
```

## 二、协议工作台左右两栏信息密度重新分配

### 现状问题

左侧聊天流里混了太多层级不同的信息：用户消息、AI 回复、诊断结论、规划摘要、前台投影入口、派发卡片、完成情况。视觉权重差不多，用户很难快速扫到关键信息。

### 设计目标

聊天流只保留对话本身，结构化信息移到右侧面板用 tab 组织。

### 具体方案

#### 左侧聊天流只保留

- 用户消息
- AI 的关键回复（诊断结论、规划摘要）
- 错误提示
- 审批 banner（见第四节）

不再在聊天流里内嵌：
- "前台投影入口"卡片组
- "派发给子 agent"卡片列表
- "子 agent 完成情况"卡片列表

这些信息改为在右侧面板对应 tab 里展示。

#### 右侧面板改为 tab 切换

```
┌─────────────────────────────────┐
│ [主机]  [任务]  [通知]           │
├─────────────────────────────────┤
│                                 │
│  (当前 tab 的内容)               │
│                                 │
└─────────────────────────────────┘
```

三个 tab：

**主机 tab**（默认）
- 当前的 Host Agents 卡片列表
- 加搜索和筛选（见第三节）
- 点击卡片打开详情抽屉

**任务 tab**
- mission 整体进度（见第六节的进度条）
- 计划摘要（原来的"前台投影入口"）
- 派发列表（原来的"派发给子 agent"）
- 完成情况（原来的"子 agent 完成情况"）
- 点击条目可以展开详情或打开抽屉

**通知 tab**
- 当前的命令通知列表
- 未读计数显示在 tab 标签上

#### tab 标签上的状态徽标

```
主机 (3⚠)     ← 3 台待审批
任务 (5/8)     ← 5/8 完成
通知 (2)       ← 2 条未读
```

这样用户不用切 tab 也能看到关键状态。

## 三、Host Agents 卡片——批量操作和筛选

### 现状问题

8 台主机还好，100 台的时候没有搜索、没有筛选、没有批量审批。运维工程师面对大量主机时最常做的操作是"只看有问题的"和"批量放行"。

### 具体方案

#### 搜索和筛选

在 Host 列表顶部加一行工具栏：

```
┌─────────────────────────────────┐
│ 🔍 搜索主机名/IP...    [筛选 ▾] │
├─────────────────────────────────┤
│ 全部(8)  待审批(1)  执行中(3)    │
│ 已完成(2) 失败(1)  排队(1)      │
├─────────────────────────────────┤
│ (卡片列表)                       │
└─────────────────────────────────┘
```

- 搜索支持主机名、IP、标签模糊匹配
- 状态筛选用 pill 按钮，点击切换，支持多选
- 筛选结果实时更新，不需要提交

#### 排序规则

默认排序优先级：
1. 待审批（最高，需要用户操作）
2. 失败（需要关注）
3. 执行中
4. 排队中
5. 已完成（最低）

同状态内按主机名排序。

#### 批量审批

当有多台主机待审批时，在列表顶部显示批量操作栏：

```
┌─────────────────────────────────┐
│ 3 台主机等待审批                  │
│ [全部批准]  [全部拒绝]  [逐个查看] │
└─────────────────────────────────┘
```

- "全部批准"一键放行所有待审批命令
- "逐个查看"打开审批队列模式，一个一个过
- 批量操作前弹确认框，显示涉及的主机和命令摘要

#### 卡片内审批按钮简化

卡片内的审批按钮从 4 个（是/授权免审/否/详情）简化为 2 个：

```
[批准 ✓]  [查看详情 →]
```

- "批准"是最常用操作，给主按钮
- "查看详情"打开审批弹窗，里面有完整选项（批准/授权免审/拒绝）
- 减少卡片内的按钮数量，降低误触风险

## 四、审批体验优化

### 现状问题

审批按钮嵌在小卡片里，视觉权重不够。审批是运维最关键的决策点，应该给更醒目的呈现。

### 具体方案

#### 全局审批 banner

当有待审批项时，在聊天流顶部（标题栏下方）显示一个醒目的黄色 banner：

```
┌─────────────────────────────────────────────────┐
│ ⚠ web-08 执行 sudo nginx -s reload 等待审批      │
│                                                   │
│ [批准]  [查看详情]              还有 2 项待审批 →   │
└─────────────────────────────────────────────────┘
```

- 固定在聊天流顶部，不随消息滚动
- 显示最紧急的一条审批
- 右侧显示剩余待审批数量，点击可以切换到下一条
- banner 背景色：待审批黄色、已批准绿色渐隐、已拒绝红色渐隐

#### 审批详情弹窗

点击"查看详情"打开弹窗，内容包括：

```
┌─────────────────────────────────────────┐
│ web-08 · 10.10.4.28                     │
├─────────────────────────────────────────┤
│ 命令全文                                 │
│ ┌─────────────────────────────────────┐ │
│ │ sudo nginx -s reload                │ │
│ └─────────────────────────────────────┘ │
│                                         │
│ 执行理由                                 │
│ 主 Agent 已将作用域收敛到当前命令。        │
│                                         │
│ 最近终端上下文                            │
│ ┌─────────────────────────────────────┐ │
│ │ $ nginx -t                          │ │
│ │ nginx: configuration file test ok   │ │
│ │ $ # 等待 reload 审批                 │ │
│ └─────────────────────────────────────┘ │
│                                         │
│ [批准]  [授权免审]  [拒绝]               │
└─────────────────────────────────────────┘
```

#### 审批反馈

审批决策后：
- banner 变色（绿色/红色）并显示结果文案，1.5 秒后自动消失或切换到下一条
- 对应的 Host 卡片状态同步更新
- 如果有下一条待审批，banner 自动切换

## 五、终端内嵌

### 现状问题

打开终端要跳到 `/terminal/:hostId` 页面，离开了工作台上下文。运维经常需要一边看 AI 的分析一边在终端里验证。

### 具体方案

#### 协议工作台的 Host 详情抽屉内嵌终端

当前 Host 详情抽屉已经有一个只读的终端输出区域。改为：

- 把只读的 `<pre>` 替换为真实的 xterm.js 终端实例
- 终端默认只读模式（显示 WorkerSession 的命令输出）
- 提供一个"接管终端"按钮，点击后切换为可交互模式
- 可交互模式下用户可以直接在终端里输入命令

```
┌─────────────────────────────────────────┐
│ web-08 · 10.10.4.28        [接管终端]    │
├─────────────────────────────────────────┤
│                                         │
│ $ nginx -t                              │
│ nginx: configuration file test ok       │
│ $ sudo nginx -s reload                  │
│ [等待审批中...]                          │
│                                         │
│ █                                       │
└─────────────────────────────────────────┘
```

#### 对话页底部终端面板

对话页（ChatPage）底部加一个可拖拽的终端面板：

```
┌─────────────────────────────────────────┐
│ 聊天消息流                               │
│                                         │
│                                         │
├─── ═══════ 拖拽调整高度 ═══════ ────────┤
│ 终端 · web-08 · 10.10.4.28    [×]      │
│ $ systemctl status nginx                │
│ ● nginx.service - A high performance... │
│ █                                       │
└─────────────────────────────────────────┘
```

- 默认收起，点击顶部导航栏的"终端"按钮展开
- 拖拽分割线调整终端高度
- 终端自动连接当前选中的主机
- 切换主机时终端自动重连
- 关闭按钮收起面板但不断开连接

#### 快捷键

- `` Ctrl+` `` 切换终端面板显示/隐藏
- `Ctrl+Shift+T` 新建终端 tab（如果需要同时连多台主机）

## 六、全局进度感知

### 现状问题

用户发起 mission 后，没有一个全局的进度指示器。进度信息散落在各个卡片里，需要自己数。

### 具体方案

#### 标题栏进度条

在协议工作台的标题栏区域（"主 Agent" 标题右侧）加一个紧凑的进度指示器：

```
主 Agent  [Fleet · web-cluster]  ████████░░ 5/8  ⚠1
```

- 进度条用颜色分段：绿色（完成）、蓝色（执行中）、橙色（待审批）、灰色（排队）
- 右侧数字显示 `完成数/总数`
- 如果有待审批，显示橙色警告图标 + 数量

#### 进度条的状态映射

```
绿色段: completed 的主机数
蓝色段: running 的主机数
橙色段: waiting_approval 的主机数
红色段: failed 的主机数
灰色段: queued 的主机数
```

#### mission 完成总结卡片

当所有主机都完成（或失败/取消）后，在聊天流里插入一张总结卡片：

```
┌─────────────────────────────────────────┐
│ ✅ Mission 完成: nginx 巡检与自修复       │
├─────────────────────────────────────────┤
│ 总计 8 台主机                            │
│ ✅ 完成 6   ⚠ 需关注 1   ❌ 失败 1       │
│                                         │
│ 耗时 4m 32s · 审批 2 次 · 命令 24 条     │
│                                         │
│ [查看详细报告]  [导出]                    │
└─────────────────────────────────────────┘
```

#### 导航栏状态同步

左侧导航的"工作台"入口也显示进度：

```
工作台
  nginx 巡检 · 5/8 ████░░
```

这样即使用户在对话页，也能一眼看到 mission 进度。

## 七、数据持久化和可追溯性

### 现状问题

运维操作需要审计。当前有 audit log（JSONL），但用户在 UI 上看不到历史 mission 的执行记录。

### 具体方案

#### mission 历史列表

在工作台页面加一个"历史 mission"入口（可以放在标题栏或右侧面板的第四个 tab）：

```
┌─────────────────────────────────────────┐
│ Mission 历史                             │
├─────────────────────────────────────────┤
│ 03-29 19:43  nginx 巡检与自修复          │
│              8 台 · 完成 · 4m 32s        │
│                                         │
│ 03-28 14:20  SSL 证书批量续期            │
│              12 台 · 完成 · 8m 15s       │
│                                         │
│ 03-27 09:10  磁盘清理                    │
│              3 台 · 失败 1 · 2m 48s      │
└─────────────────────────────────────────┘
```

#### 每个 mission 的详情页

点击历史 mission 可以查看完整记录：

- 发起人、发起时间、完成时间、总耗时
- 涉及的主机列表和每台主机的最终状态
- 每台主机的命令执行记录（命令、输出、耗时、退出码）
- 审批记录（谁审批的、什么时间、批准还是拒绝）
- AI 的规划过程摘要
- 可导出为 JSON/CSV

#### 存储方案

第一阶段不需要数据库，继续用文件：

```
<state-dir>/missions/
  ├── mission-xxx.json          ← mission 元数据 + 状态
  ├── mission-xxx-events.jsonl  ← 事件流（时间线）
  └── mission-xxx-report.json   ← 完成后的汇总报告
```

后续接入数据库时，这些文件可以作为迁移数据源。

#### 保留策略

- 默认保留最近 90 天的 mission 记录
- 超过 90 天的自动归档（压缩存储，不删除）
- 可配置保留天数：`MISSION_RETENTION_DAYS=90`

## 实现优先级

| 优先级 | 改动 | 预估工作量 | 理由 |
|---|---|---|---|
| P0 | 全局进度条 | 0.5 天 | 改动小，体验提升最直观 |
| P0 | 审批 banner + 批量审批 | 1-2 天 | 运维最高频交互 |
| P1 | 右侧面板 tab 化 | 1 天 | 解决信息密度不均衡 |
| P1 | Host 搜索和筛选 | 0.5 天 | 规模化必备 |
| P1 | 终端内嵌（抽屉内） | 1 天 | 已有 xterm.js，改造成本低 |
| P2 | 导航精简 | 1-2 天 | 涉及路由重构 |
| P2 | 对话页底部终端面板 | 1 天 | 需要拖拽分割线组件 |
| P2 | mission 历史列表 | 2 天 | 需要后端存储支持 |
| P3 | mission 详情页 + 导出 | 2-3 天 | 依赖历史列表 |
