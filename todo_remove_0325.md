# 远程主机管理任务清单 (todo_remove_0325.md)

基于 [design_remove_0325.md](/Users/lizhongxuan/Desktop/aiops-codex/design_remove_0325.md) 的需求设计，将“远程主机接入后，尽量像本地 Codex 一样做运维操作”的能力拆解为如下 6 个阶段的执行清单。

状态说明：

- `[x]` 已完成
- `[ ]` 未完成
- `（部分完成）` 表示已有前端、后端或局部链路，但还没端到端完全达标

## Phase 1: 远程主机接入与执行目标切换
> 目标：让远程主机不再只是“在线展示对象”或“附属终端”，而是聊天与终端中的默认执行目标。

- [x] 1.1 `host-agent` 主动注册到 `ai-server`，并在主机列表中展示在线状态、主机名、主机类型与基础能力标记。
- [x] 1.2 将 `ai-server` 的 gRPC 地址改为远程主机可访问地址（开发阶段采用 `AIOPS_GRPC_ADDR=0.0.0.0:18090`），打通远程主机回连链路。
- [x] 1.3 提供 Linux 侧容器化接入资产（`host-agent` Dockerfile、环境变量模板、启动方式说明），使远程主机可通过镜像方式快速接入。
- [x] 1.4 选中某台远程主机后，默认执行目标切换到该主机，不再误用本地 `server-local` 的 shell 或文件系统。
- [x] 1.5 主机切换后切断旧线程绑定或重建远程上下文，避免把前一台主机的上下文串入当前轮次。
- [x] 1.6 前端在 Host 选择弹窗中明确标注“远程可管 / 远程终端 / 只读展示”，并提供“进入终端”入口。

## Phase 2: 远程命令执行与审批主链路
> 目标：先把“远程命令管理”打稳，做到可执行、可审批、可中断、可回看。

- [x] 2.1 为远程主机线程注册专用动态工具 `execute_readonly_query` 与 `execute_system_mutation`，并通过 developer instructions 明确禁止误用本地内建 `commandExecution` / `fileChange`。
- [x] 2.2 打通 `execute_readonly_query -> ai-server -> host-agent` 的远程只读命令链路，支持检查系统状态、目录、文件、日志与服务状态。
- [x] 2.3 打通 `execute_system_mutation(mode=command)` 的远程有副作用命令链路，默认要求审批后执行。
- [x] 2.4 远程命令执行支持输出流式回传、退出码回传、失败状态回传与执行结束卡片更新。
- [x] 2.5 “停止当前任务”能够取消当前会话中的远程执行，而不只是停掉前端动画。
- [x] 2.6 将远程命令卡片的成功 / 失败 / 超时 / 取消态进一步对齐本地 Codex。
- [x] 2.7 对远程长命令增加更稳的超时与收尾策略，避免任务长期卡在“正在执行命令 / 正在整理结果”。

## Phase 3: 远程过程反馈、审批队列与错误体验对齐本地 Codex
> 目标：让远程主机执行时的可视反馈与本地 Codex 一样清晰，不让用户长时间空等。

- [x] 3.1 远程主机场景下，顶部一次只高亮一个当前状态（如 `正在浏览文件`、`正在执行命令`、`正在等待审批`），避免重复状态刷屏。
- [x] 3.2 下方聊天流插入远程轻量过程行，如：
  - `正在浏览文件`
  - `正在搜索内容`
  - `正在搜索网页`
  - `正在执行命令`
  - `正在整理结果`
- [x] 3.3 审批请求采用单点聚焦的队列机制：当前一个审批完成后，再弹出下一个，避免远程 mutation 审批扎堆。
- [x] 3.4 审批完成后，在聊天历史中保留中性的纪要卡片，如 `[已同意在 linux-01 上执行 systemctl restart nginx]`。
- [x] 3.5 对远程错误进行结构化分型，至少区分：
  - 审批未通过
  - Agent 断连
  - OS 权限不足
  - 命令退出码失败
  - 网络 / 心跳超时
  - 远程终端断开
- [x] 3.6 网络超时或远程 host-agent 断连时，前端要出现重试或恢复提醒，而不是只在内部状态里静默失败。

## Phase 4: 远程文件浏览、搜索与预览能力
> 目标：把远程“看文件 / 搜内容”从黑盒 shell 输出，升级到接近本地 Codex 的结构化体验。

- [x] 4.1 为远程主机构建协议化的文件浏览能力，支持：
  - 列目录
  - 查看文件内容
  - 搜关键字
  - 返回结构化文件元信息
- [x] 4.2 将远程文件列表从普通正文中剥离出来，作为结构化文件清单渲染，而不是整段 `find/cat` 输出直接混在正文里。
- [x] 4.3 支持点击文件名后查看文件预览，至少完成“文件名 -> 绝对路径 -> 内容预览”的单跳体验。
- [x] 4.4 远程文件操作接入 runtime 摘要与过程行，支持类似：
  - `现在浏览 /etc/nginx/nginx.conf`
  - `已浏览 3 个文件`
  - `现在搜索 keyword`
  - `已搜索 2 个位置`
- [x] 4.5 替换“`cat` 一大段黑盒输出”模式，逐步升级为结构化文件卡片、文件内容块、文件列表块与路径提示。

## Phase 5: 远程文件修改、Diff 与审批对齐本地 Codex
> 目标：让远程主机不只会“跑命令”，而是真正具备接近本地 Codex 的远程文件修改能力。

- [x] 5.1 扩展 `execute_system_mutation`，支持 `mode=file_change`，明确区分“远程命令变更”和“远程文件变更”。
- [x] 5.2 新增远程 `FileChangeApprovalCard`，在真正落盘前展示目标文件、变更原因与 diff 预览。
- [x] 5.3 新增远程 `FileChangeCard`，在审批通过并执行后展示最终 diff、状态与结果摘要。
- [x] 5.4 远程文件变更执行应支持：
  - 写入
  - 覆盖
  - 追加
  - 原子替换
  - 明确失败回滚策略
- [x] 5.5 远程文件变更结果要进入历史卡片与审计日志，确保可回看、可定位、可复盘。

## Phase 6: 安全、审计与生产化接入
> 目标：把“能连上、能执行”进一步推进到“可控、可审计、可生产部署”。

- [x] 6.1 对 `execute_readonly_query` 建立只读命令校验与基础 allowlist，避免明显的 mutation 混入只读链路。
- [x] 6.2 继续补强远程命令风险分类规则，对 `systemctl`、`docker`、`kubectl`、`git` 等命令做更精确的读写识别。
- [x] 6.3 将远程操作的审计字段落齐，至少包括：
  - `sessionId`
  - `threadId`
  - `turnId`
  - `hostId`
  - `hostName`
  - `toolName`
  - `command` / `filePath`
  - `cwd`
  - `approvalDecision`
  - `startedAt`
  - `endedAt`
  - `status`
  - `exitCode`
- [x] 6.4 提供容器化部署所需的最小资产：
  - `host-agent` Dockerfile
  - 环境变量模板
  - 启动方式说明
- [x] 6.5 将开发期的 bootstrap token 方案升级为更安全的生产方案，包括：
  - token 轮换
  - host 身份固定
  - 最小权限认证策略
- [x] 6.6 将远程接入的网络安全边界从“开发可用”升级到“生产可用”，补齐：
  - TLS / mTLS
  - 内网 / VPN 约束
  - 禁止公网裸露
  - 访问控制策略
  （已完成：新增 `HOST_AGENT_SECURITY_PROFILE=production` 启动校验，强制要求 TLS/mTLS、非默认 bootstrap token、host allowlist、来源 CIDR allowlist，并禁止 wildcard / 公网字面量 gRPC 监听；同时补充 `deploy/docker/ai-server.env.example` 与 `deploy/docker/host-agent.env.example` 作为生产接入示例）

## 最终验收清单

当以下项目全部达成时，可以认为“远程主机管理”阶段性完成：

- [x] A. 选中远程主机后，聊天默认只操作该主机。
- [x] B. 远程只读操作稳定可执行，并有清晰过程反馈。
- [x] C. 远程 mutation 默认走审批，审批后可以继续执行，不会卡死。
- [x] D. 停止任务可以真正中断远程执行。
- [x] E. 远程终端与聊天执行目标一致。
- [x] F. 远程错误不会伪装成成功。
- [x] G. 用户不会长时间面对“正在思考”而没有任何可见过程。
- [x] H. 远程文件浏览与远程文件修改都已逐步接近本地 Codex 的卡片体验。
- [x] I. 网络超时或远程断连时，会出现重试或恢复提醒。
