# 多主机协作调度控制台重设计

## 背景

这份文档用于收敛 [final-verdict.md](/Users/lizhongxuan/Desktop/aiops-codex/docs/final-verdict.md) 提出的 `orchestrator + host-agent` 方向，并替换掉之前那版“模块太多、主次不清楚”的协议页设计。

这次更新后的设计原则只有一句话：

**用户第一眼必须先看到主 Agent 对话，而不是看到一堆卡片、摘要、统计和 DAG。**

## 设计结论

之前那版页面不够好，核心问题不是信息不全，而是：

- 模块太多
- 主次不清楚
- 用户进来不知道先看哪里
- 很多辅助信息长期占着主视图
- 看起来像“系统给用户展示很多东西”，而不是“用户在指挥主 Agent 工作”

新的方向已经明确为：

1. 主 Agent 对话框是页面绝对主区
2. 右边只保留精简的 host-agent 卡片
3. 再加一条简洁的命令通知列表
4. DAG、审批细节、单机完整上下文都放弹窗或抽屉
5. 能少的模块就少，能弹出的就不要常驻

还有一条这次已经明确拍板：

**主 Agent 对话框本身应当和普通会话页几乎一模一样，只是在会话流里插入两类 UI 卡片。**

这两类卡片是：

- 主 Agent 发给子 agent 的极简任务条
- 子 agent 完成情况的极简结果条

## 页面定位

这个页面不是“大而全的总控大盘”，而是：

`主 Agent 工作台`

页面的核心体验应该是：

- 用户给主 Agent 提需求
- 主 Agent 分析情况
- 主 Agent 生成计划
- 主 Agent 把任务派发给 host-agent
- 用户在右侧快速看到哪些主机在工作、哪些需要审批
- 如果要深挖，再点开详情

也就是说，这个页面首先是“对话驱动的任务工作台”，其次才是“多主机状态面板”。

## 总体布局

推荐布局改成非常明确的两栏：

1. 左侧可收缩导航
2. 中间主区 `Main Agent Conversation`
3. 右侧侧栏 `Host Cards + Command Notifications`

不再做三栏常驻主界面。

### 1. 左侧导航

左侧菜单栏必须支持收缩。

收缩后保留：

- 图标
- 当前激活态
- 设置入口

展开后再显示：

- 文本标签
- 当前会话标题
- 二级说明

原因很简单：

- 运维工作台需要尽量把横向空间留给主对话区
- 长期展开的大侧栏会让主页面显得拥挤

### 2. 中间主区

中间区域是页面绝对重点，宽度要明显大于右侧。

这里应承载：

- 用户和主 Agent 的对话
- 主 Agent 的分析结果
- 主 Agent 的计划摘要
- 主 Agent 下发给子 agent 的任务卡片
- 对主 Agent 的继续追问和补充限制

### 3. 右侧侧栏

右侧不是信息超市，只保留两块：

1. `Host Agents`
2. `命令通知`

并且都要尽量做薄、做短、做可扫。

## 明确删除或降级的内容

下面这些内容不应该继续作为主页面常驻模块：

- `Mission Control` 大标题
- “多主机协作调度控制台”大 Hero 文案
- “左栏盯任务拆分...” 这类解释说明
- 顶部按钮：`新建任务` / `打开任务回放` / `暂停编排`
- “当前任务”大卡片
- 长期常驻的 DAG 大面板
- 长期常驻的大统计区
- 常驻的 `Mission Summary Card`
- 常驻的 `Planner Brief` 大块结构卡

这些东西不是完全不要，而是：

- 要么删掉
- 要么放进对话内容里
- 要么放进弹窗里
- 要么在详情里看

## 页面主模块

最终主页面只保留这 3 个核心模块：

### 1. Main Agent Conversation

这是主模块，占页面大头。

它的视觉和交互应尽量复用普通聊天页：

- 同样的消息流
- 同样的滚动方式
- 同样的底部输入框
- 同样的会话阅读节奏

不能再单独设计成另一套“像聊天但其实不是聊天”的中控面板。

这里展示：

- 用户消息
- 主 Agent 回复
- 主 Agent 分析
- 主 Agent 生成的计划
- 主 Agent 派发给子 agent 的任务卡片

这里不展示：

- 原始终端输出
- 大量主机硬件指标
- 整个 DAG 画布
- 过多统计数字

### 2. Host Agents

右侧第一块模块，展示当前真正工作的 host-agent。

卡片要尽量压小，普通状态下最好只有两行：

- 主机名
- IP
- 当前状态
- 当前阶段和最新命令合并成一行摘要

如果是审批态，再额外显示：

- 审批按钮：`是 / 授权免审 / 否`
- `详情`

不要在卡片里继续堆：

- 大段摘要
- 多行资源指标
- 太多按钮
- 很长的终端输出

细节一律放详情抽屉。

### 3. 命令通知

右侧第二块模块，专门显示最近的命令执行通知。

建议格式非常直接：

- `127.0.0.3 执行 sudo nginx -s reload 已完成`
- `10.10.4.28 执行 sudo nginx -s reload 等待审批`
- `10.10.4.29 执行健康探针，当前 3 / 5 已完成`

这块的作用是让用户扫一眼就知道最近发生了什么，而不是看详细日志。

## 主 Agent 对话区设计

主 Agent 对话区应该从上到下只有两部分：

1. `Conversation Stream`
2. `Composer`

### Conversation Stream

消息类型建议：

- 用户需求
- 主 Agent 分析回复
- 主 Agent 规划回复
- 主 Agent 派发卡片
- 系统性回执

其中“派发卡片”非常重要。

它应该直接出现在主对话流里，而不是另做一个大模块。

“子 agent 完成情况卡”也一样，应该插在同一条会话流里，而不是放成一个和聊天平级的大区域。

卡片内容建议精简成：

- 一行就够
- 主机名
- 一句任务或结果摘要
- 当前状态
- 点击后看详情

这样用户在看对话的时候，就能顺带看到：

- 主 Agent 说了什么
- 它决定怎么拆任务
- 它把任务分给了哪些 host-agent
- 哪些 host-agent 已经完成或回报了什么结果

一句话：

**用户看到的是“同一个会话，只是多了任务卡和完成卡”，而不是“聊天区旁边又长出一个新的任务系统”。**

### Composer

输入框是用户继续给主 Agent 发指令的地方。

这里保留少量辅助动作即可：

- 发送
- 插入经验包
- 追加主机范围

不要堆太多二级按钮。

## DAG 设计

DAG 不是不要，而是降级成辅助信息。

新的规则：

- 主页面不再常驻显示 DAG
- 用一个按钮打开 DAG 弹窗
- 只有用户主动想看任务拆分时才展开

### DAG 弹窗里展示什么

- 任务拆分节点
- 节点依赖关系
- 当前执行状态
- 哪个节点在等待审批

### DAG 弹窗里不需要展示什么

- 过多文字解释
- 太多统计面板
- 和主机详情重复的内容

一句话：

**DAG 是辅助理解任务拆分的，不应该抢走主 Agent 对话的注意力。**

## Host Agent 卡片设计

host-agent 卡片要从“厚卡片”改成“薄卡片”。

### 正常态

建议只显示两行主信息：

- 主机名
- IP
- 状态 pill
- 当前阶段和最新命令合并成一行

### 审批态

审批态是唯一可以比正常态多一点内容的情况。

需要显示：

- 主机名
- IP
- 待审批命令预览
- `是`
- `授权免审`
- `否`
- `详情`

并保持：

- 命令尽量压成 1 行，最多 2 到 3 行
- 仍然是一张薄卡
- 不在卡片里塞入过多上下文

### 详情抽屉

点击卡片后才打开主机详情。

详情抽屉里再展示完整信息：

- 主机基本信息
- 当前命令
- 终端输出
- 历史命令
- 主 Agent 和该 host-agent 的详细会话

也就是说：

- 主页面负责“扫”
- 详情抽屉负责“看深”

## 审批设计

审批要遵循一个原则：

**默认就地处理，细看再开弹窗。**

### 默认路径

用户在 host-agent 卡片上直接处理：

- `是`
- `授权免审`
- `否`

### 深度路径

只有点击 `详情` 时，才打开审批弹窗。

审批弹窗里展示：

- 命令全文
- 当前命令为什么需要审批
- 最近终端上下文
- 再次确认按钮

### 审批作用域

`授权免审` 只对当前命令生效。

不能扩散到：

- 当前主机后续命令
- 当前任务其它主机
- 用户级长期策略

## 命令通知列表设计

命令通知列表要尽量像“事件流”，而不是详细日志流。

建议每条通知只保留：

- 时间
- IP
- 命令动作
- 当前结果

状态类型建议：

- 已完成
- 执行中
- 等待审批
- 已拒绝
- 排队中

不要在通知里放：

- 原始终端输出
- 多行异常栈
- 太长的命令全文

## 主机详情抽屉

主机详情抽屉是页面里的主要“深挖入口”。

布局建议：

- 左边：终端工作区
- 右边：主 Agent 与该 host-agent 的对话

顶部再显示：

- 主机名
- IP
- OS
- CPU
- 内存
- 磁盘
- agent 版本

这个抽屉解决的是：

- 为什么在这台机器上执行这条命令
- 这条命令执行到了什么程度
- AI 之前尝试过什么
- 当前上下文是否足够支持审批或判断

## 交互优先级

新的优先级必须非常清楚：

1. 先看主 Agent 对话
2. 再看右侧哪些主机在工作
3. 再扫最近命令通知
4. 想看任务拆分时再打开 DAG
5. 想看具体上下文时再点开主机详情

这和之前那版最大的不同是：

- 不要求用户一进来就理解整个系统全貌
- 先让用户顺着主 Agent 工作流往下看

## 页面风格要求

这页必须克制，不要做成“满屏模块”。

明确要求：

- 保持大面积留白
- 减少大标题和解释文案
- 模块数量能少则少
- 卡片尽量薄
- 非核心信息移到弹窗
- 不做大屏监控风

用户不应该感受到“系统很复杂”，而应该感受到“我只需要和主 Agent 对话，其它辅助信息都在旁边待命”。

## 后续继续优化的方向

后面如果继续收，还建议做这几件事：

1. 继续减少主页面里的标签和解释性文字
2. 让主 Agent 派发卡片再短一点
3. 让 host-agent 卡片高度再压一点
4. 命令通知列表继续往“单行可扫”方向收
5. DAG 弹窗默认尺寸再克制一点

## 最终结论

多主机协作调度控制台不应该是“把所有能力铺满一页”。

它应该是：

- 一个主 Agent 对话工作台
- 右边挂着精简主机卡片
- 再带一条简单命令通知流
- 其余复杂内容全部弹出查看

只有这样，用户才不会觉得页面看不懂、模块太多、东西都挤在一起。
