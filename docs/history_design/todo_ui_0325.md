# AIOps Codex UI 任务拆解清单

基于对 Codex App 一致性体验的深度定制需求，将新增的 8 项产品级特性整理为如下第 5 阶段（Phase 5）可执⾏的任务检查表。

状态说明：
- `[x]` 已完成
- `[ ]` 未完成
- `（部分完成）` 表示已有前端或局部实现，但还没端到端打通

## Phase 5: 深度交互与 Codex App 体验对齐

### 1. 全局设置面板（额度、模型、推理等级）
- [ ] 1.1 在左下角 `SettingsIcon` 点击弹出的 Modal（或右侧悬浮面板）中新增“账户与模型设置”页面。
- [ ] 1.2 对接 `codex app-server` API（例如 `GET /api/v1/settings`），获取并展示当前账户剩余额度。
- [ ] 1.3 提供“模型选择”下拉列表（如切换底层驱动大模型及版本）。
- [ ] 1.4 提供“推理强度”(Reasoning Intensity) 设置项（如 Low / Medium / High 档位切换），并支持变更后持久化保存。

### 2. 回复气泡一键复制
- [ ] 2.1 在 `MessageCard` (仅对 Assistant 角色的正文及代码块组件) 附加独立的 “复制 (Copy)” 悬浮图标按钮，位于底部或右上角。
- [ ] 2.2 交互优化：正常态隐蔽，鼠标 Hover 时渐显；实现纯净本文内容的剪贴板写入。
- [ ] 2.3 操作态反馈：点击后图标状态切换，并浮现明显的 `✓ 复制成功` 临时反馈动画（常驻 2~3 秒后复原）。

### 3. 文件名高亮与悬浮绝对路径
- [ ] 3.1 改造前端聊天流对于文件结构链接（如 `[App.vue](/Users/xxx/App.vue)` 格式的 Markdown Link）的渲染解析器。
- [ ] 3.2 将命中规则的文件名渲染成具有强感官辨识度的蓝色胶囊/可交互 Text 样式，不作为标准的外部跳转链接。
- [ ] 3.3 完善原生 `title` 属性或引入定制化的 Tooltip 组件，使得鼠标悬浮在蓝色文件实体名上方时，清晰透出文件的绝对路径完整信息。

### 4. Plan 规划栏缝合式固定 (Dock Posture)
- [ ] 4.1 对当下的 `runtime-plan-dock` 容器及其内部嵌套的 `PlanCard(compact)` 组件的 CSS 间距进行二次校准。
- [ ] 4.2 消除底部的所有微缝外边距，移除组件下半部的独立边框与下沿圆角。
- [ ] 4.3 调整过渡阴影（`box-shadow`）等参数，使其在视觉上一体化，彻底做成宛如“从底部输入框 `Omnibar` 抽出来的拓展托盘”。

### 5. 审批工作台 (Auth Overlay) 霸屏重构
> **目标**：彻底将需要阻断和审批的卡片界面从聊天流（Chat Stream）中剥离出去，降维接管输入区。
- [ ] 5.1 **拦截不刷屏**：遇到 `waiting_approval` 的 Auth 卡片时停止其向 `store.snapshot.cards` 时间线的传统堆叠。
- [ ] 5.2 **输入区原位接管**：在其原本的 `Omnibar` 操作栏位置之上，渲染审批面板 UI（原有的“同意”、“拒绝”、“记忆”选项及环境选择下拉），大小跟输入框一样大,视觉上完全盖住输入框。
- [ ] 5.3 **降级与折叠能力**：审批面板额外提供“折叠”按键能够收缩或退让出文本输入区，以支撑用户直接打字拒绝并说“换个方式执行”。
- [ ] 5.4 **后置执行留痕**：用户在审批面板提交抉择后，强制销毁悬浮面板并在已凝固的上方聊天流历史中，推入一条独立清晰的已操作记录（如单纯一句中性的 `[已同意执行 npm run build]` 纪要卡片），还原操作上下文。

### 6. 运行状态树 (Timeline Status) 补充及底层协议约束
> **目标**：位置沿用当前流底部和活动指示器区域不变；主要核心在于对齐前后端的行为协议边界。以下主要作用于 `codex app-server` 的底层行为。
- [ ] 6.1 前端确保顶部只高亮**一个**动名词组成的“当前状态”（如：`浏览中`、`分析中`、`修改中`），且同一时间段不接受并无视后端发来的连续重复状态指令。
- [ ] 6.2 下方的瀑布时间线承接并收归过往累加的完结步骤，转化展现为极简的中文字符缩略表述结果态（例如 `已浏览 3 个文件`、`已看完`，限制中文字符宽说明 `detail` 不超过 18 个字内）。
- [ ] 6.3 严格约束后端触发 `browsing` 和 `browsed` 来包裹并对齐查看目录、读文件与代码浏览系列动作。
- [ ] 6.4 严格约束后端触发 `searching` 和 `searched` 来包裹查关键词、引用及 web search。
- [ ] 6.5 严格约束后端触发 `editing` 和 `edited` 来首尾封皮修改文件的编辑意图操作。
- [ ] 6.6 严格约束后端触发 `executing` 和 `executed` 来包装命令、编译构建与跑脚本等任务操作。
- [ ] 6.7 严格约束后端触发 `testing` 和 `tested` 作为质量把控验证环节的起始边界发号器。
- [ ] 6.8 严格把控只有面临真正需要用户干预打断的时候（非 auto_accept 场景），才抛出确切的单次 `waiting_approval` 或 `waiting_confirmation` 动作申请。
- [ ] 6.9 要求系统架构在解决和彻底完成全盘派发任务规划后的最终尾节点，且只有这一次，才发送全局的 `completed` 终结信号。

### 7. 聊天容器底端留白优化 (Chat Container Spacing)
- [ ] 7.1 进一步调大对话内容区域（`.chat-container`）的底部内边距（`padding-bottom`），使得最末端的对话气泡与下方固定的输入框（及 Plan 栏）之间额外预留出宽裕的呼吸空间，提升视觉阅读体验。

### 8. 终端展示与审批队列优化
- [ ] 8.1 优化 `TerminalCard` 的尺寸：限制黑底终端的最大高度（例如 `max-height` 设为 `200px` 或 `250px`），对于输出较少的过程按需撑开，内部超出则显示滚动条，避免简单命令占据过大版面。
- [ ] 8.2 引入“审批队列（Approval Queue）”机制：避免多个审核 UI 一次性扎堆弹出；采用排队方式，当前一个审核点完（同意/拒绝）后，再弹出下一个审核，维持单点聚焦体验。
