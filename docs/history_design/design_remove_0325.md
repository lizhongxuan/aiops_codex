# design_remove_0325

## 0. 文档定位

本文是“远程主机管理能力”专项设计文档。

它不是要再发明一套独立的远程运维产品，而是要把当前项目已经具备的：

- 浏览器对话 UI
- `ai-server`
- `codex app-server`
- `host-agent`

缝合成一条完整链路，让用户在选中一台远程主机后，获得尽量接近“本地使用 Codex 做运维”的体验。

本文重点回答 5 件事：

- 远程主机管理的最终体验到底应该长成什么样
- 当前项目已经做到哪里，还差哪些能力
- 远程主机执行链路应该如何分层
- 审批、安全、审计边界怎么定
- 接下来应按什么顺序落地

本文覆盖：

- 远程主机对话执行语义
- 远程只读、变更、终端、文件操作能力
- 前端卡片与状态反馈要求
- `ai-server / codex app-server / host-agent` 的职责边界
- Docker 化接入与运行要求

本文不覆盖：

- 多租户 RBAC
- TLS / mTLS 的最终生产级安全方案
- 审计日志入库与报表
- K8s / CMDB / Ansible 等外部系统的深度集成

## 1. 先说结论

远程主机管理的目标可以定成一句话：

> 用户在页面里选中某台远程主机后，Codex 的默认执行目标就切到这台主机，后续的浏览文件、查配置、跑命令、改文件、审批、进入终端、停止任务，都应尽量对齐本地 Codex 的体感，只是操作对象从本机变成远程 Linux。

这句话可以再拆成 6 条产品级结论：

1. 远程主机不是“附加终端”，而是完整的默认执行目标。
2. 聊天执行和终端执行必须指向同一台主机，不能出现 UI 上选的是远程机，实际还在本地运行的情况。
3. 远程只读操作和远程变更操作必须物理分层，前者快，后者严。
4. 用户应先看到过程，再等待结果，不能长时间只显示“正在思考”。
5. 远程文件浏览和远程文件修改不能永远退化成黑盒 shell，最终要补齐接近本地 Codex 的文件卡片体验。
6. “体验对齐本地 Codex”不等于“协议照搬本地 Codex”，而是用户感知上的一致性：可见、可控、可审批、可中断、可回看。

## 2. 当前状态与差距

### 2.1 当前已经具备的能力

当前项目已经有以下基础：

- `host-agent` 可以主动注册到 `ai-server`
- 远程主机可以在前端主机列表中显示在线
- 页面已经支持进入远程主机终端
- `ai-server` 已开始为远程主机注册动态工具：
  - `execute_readonly_query`
  - `execute_system_mutation`
- 远程命令已经可以从聊天链路下发到 `host-agent`
- 有副作用的远程命令已经可以走审批

这说明系统已经不再是“只能展示远程主机在线状态”，而是进入了“远程命令管理”的阶段。

### 2.2 当前仍然不足以称为“像本地 Codex 一样”

目前距离目标还差 4 个关键层面：

1. 远程文件系统能力还不完整  
   现在更多还是远程跑命令，远程 `file_change`、diff、文件预览、结构化文件浏览还不完整。

2. 远程浏览过程反馈还不够协议化  
   例如“正在浏览文件 / 已浏览 3 个文件 / 正在搜索网页 / 正在页内查找”这些反馈，当前有部分能力，但还未完全远程协议化。

3. 远程主机上下文还不够强约束  
   最终应做到选中远程主机后，Codex 默认只操作这台主机；当前虽然已经开始约束，但还需要继续收紧与补全。

4. UI 体验仍有本地与远程差异  
   本地链路和远程链路都能执行，但卡片密度、审批流、文件卡片、错误呈现、过程摘要等细节还需要进一步统一。

## 3. 最终体验定义

### 3.1 主机选择语义

当用户在页面中选择某台主机后，应遵循以下规则：

- 当前对话轮次的默认执行目标就是该主机
- 聊天中的工具调用、命令执行、文件操作默认都作用于该主机
- 顶部 Host Selector、终端页、审批卡、命令卡都必须显示当前目标主机
- 若用户切换主机，应切换到新的线程上下文或清空旧线程绑定，避免把上一个主机的上下文串入本轮

换句话说：

- “主机选择”不是视觉筛选条件
- 它是执行语义边界

### 3.2 用户希望看到的行为

从用户角度，远程运维应至少满足这些场景：

1. 我选中一台 Linux 主机后，直接问：
   - “看下 CPU 情况”
   - “帮我检查 nginx 是否运行”
   - “打开 `/etc/nginx/nginx.conf` 看看”
   - “帮我把这个配置改掉然后重启服务”

2. Codex 的反馈应像本地一样：
   - 先有轻量过程提示
   - 再出现文件浏览、命令执行、审批、结果总结
   - 长时间任务支持停止
   - 失败能看错误输出，不会假装成功

3. 若动作涉及风险：
   - 会弹审批
   - 审批可以排队
   - 批准后继续执行
   - 拒绝后 Codex 能继续换方案，不会卡死

4. 若要进入终端：
   - 能直接进入该远程主机的终端
   - 聊天里执行的命令与终端页看到的是同一台机器

## 4. 分层架构定版

## 4.1 职责边界

建议把这套系统固定为四层：

### 浏览器

职责：

- 展示聊天时间线、终端、审批卡、文件卡
- 提供主机切换、审批决策、停止任务、进入终端等交互
- 通过 HTTP + WebSocket 与 `ai-server` 通信

### ai-server

职责：

- 作为业务编排层
- 维护浏览器会话状态、线程状态、主机状态
- 启动并维护 `codex app-server`
- 把前端请求转成 `codex app-server` JSON-RPC
- 把 `codex app-server` 的事件流归一化为前端卡片与运行时状态
- 把远程命令与远程文件能力下发给 `host-agent`
- 负责审批、审计、状态机与远程执行回传

### codex app-server

职责：

- 维护 Agent Runtime 和线程
- 决定该读文件、搜网页、问审批、跑工具还是给最终答复
- 通过动态工具调用 `ai-server` 暴露的远程能力

### host-agent

职责：

- 作为远程主机执行面
- 负责在目标 Linux 主机上执行命令、打开终端、操作文件
- 主动回连 `ai-server`
- 返回输出、退出码、错误、超时、文件结果

## 4.2 基本原则

远程主机管理采用“编排在 `ai-server`，执行在 `host-agent`”的模式。

不建议：

- 让浏览器直接 SSH 远程主机
- 让 `codex app-server` 直接连接远程主机
- 把远程主机逻辑硬塞进本地 `commandExecution`

推荐链路是：

浏览器 -> `ai-server` -> `codex app-server` -> 动态工具 -> `ai-server` -> `host-agent`

## 5. 远程能力清单定版

要想做到“像本地 Codex 一样”，远程能力至少要拆成 5 类。

### 5.1 远程只读查询

用途：

- 看系统状态
- 看目录
- 看文件
- 搜日志
- 查服务状态

典型命令：

- `uname`
- `uptime`
- `df -h`
- `free -m`
- `systemctl status`
- `journalctl`
- `cat`
- `sed -n`
- `tail`
- `find`
- `grep`

实现要求：

- 通过 `execute_readonly_query` 走远程链路
- 必须有 allowlist 或只读规则校验
- 默认不需要审批
- 前端要有“正在浏览 / 已浏览 / 正在搜索 / 已搜索”过程反馈

### 5.2 远程命令变更

用途：

- 安装软件
- 重启服务
- 修改权限
- 删除文件
- 发信号
- 运行有副作用脚本

实现要求：

- 通过 `execute_system_mutation(mode=command)` 走远程链路
- 默认需要审批
- 审批通过后再执行
- 命令成功后收敛成摘要
- 命令失败时保留完整输出
- 必须支持取消 / 中断

### 5.3 远程文件浏览与搜索

用途：

- 列目录
- 查看文件内容
- 搜关键字
- 展示文件路径与文件名

实现要求：

- 不能永远退化成“`cat` 一大段黑盒输出”
- 应逐步补成结构化文件浏览能力
- 前端要支持：
  - 文件名高亮
  - hover 绝对路径
  - 文件列表与普通正文分离
  - 未来支持点击文件名打开文件预览

### 5.4 远程文件修改

用途：

- 修改配置文件
- 生成脚本
- 变更 systemd / nginx / app 配置

实现要求：

- 最终需要支持 `execute_system_mutation(mode=file_change)`
- 要有 `FileChangeApprovalCard`
- 要有 `FileChangeCard`
- 要有 diff
- 审批通过后才真正落盘

这是实现“远程主机像本地 Codex”最关键、也最晚补齐的一环。

### 5.5 远程终端

用途：

- 用户想手动继续排查
- 聊天执行之外，需要进入交互式 shell

实现要求：

- 终端页必须绑定到远程 host
- 聊天命令与终端页面要共享相同主机语义
- 长任务支持中断
- 终端会话结束后及时回收

## 6. UI 与交互要求

### 6.1 过程反馈要对齐本地 Codex

用户发出请求后，不能长时间空等。

远程主机场景下，至少要有以下过程态：

- `正在浏览文件`
- `正在搜索内容`
- `正在搜索网页`
- `正在执行命令`
- `正在等待审批`
- `正在整理结果`

同时还要有摘要态：

- `已浏览 3 个文件`
- `已搜索 2 次网页`
- `已处理 1 个命令`
- `已修改 2 个文件`

原则：

- 顶部一次只高亮一个当前态
- 下方聊天流保留轻量过程行
- 不重复、不刷屏、不出现连续冗余文案

### 6.2 审批体验

远程 mutation 审批必须满足：

- 单点聚焦
- 队列排队
- 当前一个审批处理完，再弹下一个
- 用户拒绝后可以继续输入“换个方式做”
- 审批完成后，聊天流中应保留中性纪要

### 6.3 命令卡片

远程命令卡片行为与本地一致：

- 运行中显示终端输出
- 成功后收敛成一行摘要
- 失败保留完整输出
- 不能把长命令撑出屏幕
- 对 `python / bash / zsh / sh / shell` 等命令要做轻量类型识别与折叠显示

### 6.4 错误卡片

远程错误不能全部揉成一类。

至少要区分：

- 审批未通过
- Agent 断连
- OS 权限不足
- 命令退出码失败
- 网络 / 心跳超时
- 远程终端断开

用户需要看到的是：

- 失败原因
- 是否可重试
- 是否建议换方式

而不是单纯的底层 RPC 文本。

## 7. 安全与审计边界

### 7.1 只读与变更必须分层

建议固定规则：

- 只读查询：
  - 快速执行
  - allowlist
  - 默认不审批

- 变更操作：
  - 默认审批
  - 审批可记忆当前会话
  - 必须有 host、命令、cwd、reason 记录

### 7.2 远程主机最小信任模型

当前阶段建议采用：

- `host-agent` 主动回连 `ai-server`
- 用 bootstrap token 做最小认证
- gRPC 不直接暴露公网
- 推荐跑在：
  - Tailscale
  - WireGuard
  - 同一内网 + 防火墙

生产前还应补：

- TLS / mTLS
- token 轮换
- host 身份固定
- 审计日志落库

### 7.3 审计字段

每次远程操作至少要记录：

- `sessionId`
- `threadId`
- `turnId`
- `hostId`
- `hostName`
- `operator`
- `toolName`
- `command` 或 `filePath`
- `cwd`
- `approvalDecision`
- `startedAt`
- `endedAt`
- `status`
- `exitCode`

## 8. Docker 化接入要求

### 8.1 Linux 侧部署方式

推荐优先使用容器。

原因：

- 发布简单
- 升级方便
- 依赖清晰
- 和宿主业务进程解耦

推荐形态：

- 一台 Linux 主机跑一个 `host-agent` 容器
- 容器通过环境变量连接到 `ai-server`

### 8.2 最低配置要求

`host-agent` Docker 镜像至少应满足：

- 能运行 `host-agent` 二进制
- 有基础 shell 环境，如 `sh/bash`
- 支持 pseudo-TTY
- 能回连 `ai-server`

环境变量至少包括：

- `AIOPS_SERVER_GRPC_ADDR`
- `AIOPS_AGENT_HOST_ID`
- `AIOPS_AGENT_HOSTNAME`
- `AIOPS_AGENT_BOOTSTRAP_TOKEN`

### 8.3 ai-server 暴露要求

为了让 Linux 容器能连到你电脑上的 `ai-server`，gRPC 必须监听可达地址。

当前建议：

- `AIOPS_GRPC_ADDR=0.0.0.0:18090`

但这只是开发阶段方便接入的方案。

生产建议：

- 不直接暴露公网
- 只绑内网 / VPN 地址
- 配置额外访问控制

## 9. 分阶段落地建议

### Phase A：远程命令管理可用

目标：

- 远程主机上线
- 远程只读命令可跑
- 远程变更命令可审批
- 远程终端可进入

验收：

- 用户选中一台 Linux 主机后，可以问“看下 CPU/磁盘/服务状态”
- 能看输出
- 有风险命令会审批

### Phase B：远程主机体验对齐本地

目标：

- 文件浏览与文件搜索过程反馈完整
- 审批队列稳定
- 停止任务可中断远程执行
- 命令成功/失败卡片行为统一
- UI 过程摘要稳定

验收：

- 用户主观感受已经接近本地 Codex 的操作节奏

### Phase C：远程文件修改对齐本地

目标：

- 支持结构化远程 `file_change`
- 支持 diff 与审批卡
- 支持更强的文件预览与结果回看

验收：

- 用户可以像本地 Codex 一样，让系统修改远程配置文件并审批确认

## 10. 最终验收标准

如果下面 8 条都满足，可以认为“远程主机管理”阶段性达标：

1. 选中远程主机后，聊天默认只操作该主机。
2. 远程只读操作无需多余绕路，能稳定执行并有过程反馈。
3. 远程 mutation 默认走审批，审批后能继续执行，不会卡死。
4. 停止任务可以真正中断远程执行，不只是停前端动画。
5. 远程终端与聊天执行目标一致。
6. 远程错误不会伪装成成功。
7. 用户不会长时间面对“正在思考”而无过程反馈。
8. 远程文件浏览与远程文件修改都能逐步接近本地 Codex 的卡片体验。
9. 网络超时,会出现重试的提醒

## 11. 本文结论

当前项目下一阶段的目标，不应该再停留在“让远程 Linux 连上来”，而应该正式转成：

> 把远程主机做成 Codex 的默认工作对象，而不是一个附属终端。

具体落地顺序建议固定为：

1. 先把远程命令管理链路彻底打稳
2. 再补远程文件浏览 / 搜索 / 预览
3. 最后补远程 `file_change`，把远程体验真正拉到接近本地 Codex

这三步做完，用户才会真正觉得：

“我选了一台远程主机之后，用起来就像在本地用 Codex 一样做运维。”
