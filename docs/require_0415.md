# 运维 AI Chat 需求改造清单（2026-04-15）

## 1. 文档目标

本清单用于将当前 `aiops-codex` 的 Chat / Workspace 能力，从“能对话的运维助手”继续改造成“可观测、可审计、可中断、受约束的 Incident 操作台”。

原则：

- 不推翻现有架构，优先复用已有的 `Approval`、`Audit`、`Workspace`、`Host Agent`、`Coroot UI Card`、`Agent Loop` 基础能力。
- 保留原始参考里真正正确的方向，同时吸收新增的高价值改造项，形成可落地的产品与工程清单。
- 默认面向生产环境，优先考虑可控性、透明度、回放能力和人机协同，而不是“更像人”的聊天体验。

## 2. 本次必须保留的核心方向

以下 5 类能力继续作为底座要求，不能在改造过程中被弱化：

- [ ] **状态流式反馈（Streaming Observability）**：Agent 执行时必须持续输出阶段、动作、结果、阻塞态，而不是只给最终答案。
- [ ] **人机协同与安全拦截（HITL + RBAC）**：高风险动作必须可审批、可拒绝、可追踪，且权限边界继承用户身份。
- [ ] **Server-Driven UI / 结构化卡片**：指标、日志、拓扑、审批、回滚建议不能只靠 Markdown 文本承载。
- [ ] **异步任务与可打断性**：长任务必须脱离同步请求模型，支持后台执行、进度推送与强制停止。
- [ ] **动态上下文挂载**：模型每轮推理前都应能自动挂载实时环境快照、当前告警态和相关 SOP / Runbook。

## 3. 本次新增且必须纳入需求的高价值修改

这是相对原始参考新增、且应正式写入需求的内容：

- [ ] **证据优先**：所有结论必须绑定证据来源，不允许无证据诊断。
- [ ] **计划先行、执行后置**：默认先给排查/修复计划，再进入执行；执行前要明确风险、影响面和回滚路径。
- [ ] **受约束动作系统**：高风险操作不能依赖自由生成 shell，应优先使用白名单工具、schema 校验、dry-run 和 blast radius 评估。
- [ ] **完整审计与可回放**：要能事后回放“谁、何时、基于什么证据、执行了什么动作、谁审批的、结果如何”。
- [ ] **拓扑与环境感知**：Agent 需要理解服务依赖、主机/集群/租户边界、环境隔离和变更影响范围。
- [ ] **执行后自动验证**：动作完成后必须自动检查指标、日志、告警是否恢复，必要时提示回滚。
- [ ] **不确定性表达**：允许 Agent 明确表达置信度、候选假设和下一步最有效的缩小范围动作。
- [ ] **双模式交互**：产品层必须显式区分“分析模式”和“执行模式”。
- [ ] **面向 Incident 而不是单轮问答**：对话要有时间线、证据流、审批流、待办和交接摘要。
- [ ] **结构化后续动作**：每个结论后应给出可点击的下一步，而不是让用户反复手写提示词。

## 4. 产品定位改造

### 4.1 目标定位

- [ ] 将 Chat 从“对话框”升级为“受控的运维工作台”。
- [ ] 将 Workspace 从“多主机编排视图”升级为“Incident 协同视图”。
- [ ] 将 Approval 从“命令是否执行”升级为“计划、动作、风险、回滚、验证”的统一门控。

### 4.2 默认交互原则

- [ ] 默认进入 `分析模式`，只读查询免审批，禁止隐式变更。
- [ ] 进入 `执行模式` 前，必须先展示计划摘要、目标范围、风险、预期影响、回滚建议。
- [ ] 任何危险动作必须可中断，且停止后前端能看到中断结果与残留状态。
- [ ] 若存在结构化卡片可承载结果，则不退化成纯文本大段总结。

## 5. P0 改造清单

P0 表示没有这些能力，就不应把产品定义为“体验优秀的运维 AI Chat”。

### P0-1 多轨状态流与阶段可观测

- [ ] 对每次运行输出统一阶段：`理解问题 -> 制定计划 -> 取证 -> 分析 -> 待审批 -> 执行中 -> 验证中 -> 完成/中止/失败`
- [ ] 每个阶段支持附带结构化事件：工具名、目标对象、耗时、结果条数、错误摘要。
- [ ] 前端主线程只展示高价值进度，详细过程进入 process fold / timeline / evidence 面板。
- [ ] 阻塞态（审批、等待用户、后台任务、停止中）必须和正文解耦显示。

验收标准：

- [ ] 用户发起“排查订单服务延迟升高”，5 秒内能看到明确阶段推进，而不是空转。
- [ ] 工具调用失败时，线程里能看到失败发生在哪个阶段、针对哪个工具、失败原因是什么。

### P0-2 证据链优先

- [ ] 每条关键结论必须能挂载证据引用：日志片段、PromQL、Trace、主机命令输出、配置 diff、变更记录。
- [ ] 证据与结论支持双向跳转：从结论点进证据，从证据回看引用它的结论。
- [ ] 当证据不足时，Agent 必须输出“当前假设”而不是伪装成确定结论。

验收标准：

- [ ] 任意一个“根因判断”都能展开看到至少一个直接证据源。
- [ ] 无证据时，回答中明确标注“假设/待验证”，而不是直接下结论。

### P0-3 分析模式 / 执行模式 + 计划前置

- [ ] 前端显式展示当前模式：`分析模式` 或 `执行模式`。
- [ ] 默认所有新会话先进入分析模式。
- [ ] 进入执行模式前必须生成计划卡，包含：目标、步骤、风险、影响范围、回滚建议、验证方案。
- [ ] 计划卡审批通过后，Agent 才能发起变更工具或危险命令。

验收标准：

- [ ] 用户说“直接帮我修”，系统也必须先给计划，再允许执行。
- [ ] 未通过计划审批前，任何 mutation 工具都不能落地执行。

### P0-4 身份透传与细粒度权限边界

- [ ] Agent 访问 K8s / AWS / 主机 / MCP 时必须继承当前用户身份或其受限代理身份。
- [ ] 严禁使用平台级 root token 直接替代用户执行生产变更。
- [ ] 审批记录必须包含发起人、审批人、执行身份、目标资源。

验收标准：

- [ ] 同一个动作，不同权限用户看到的可执行范围不同。
- [ ] 事后审计能明确区分“用户无权执行”与“模型判断不执行”。

### P0-5 危险动作断点与受约束执行

- [ ] 所有工具定义增加风险元数据：`readonly / mutation / dangerous / requires_plan / requires_approval / rollback_supported`。
- [ ] 高风险工具进入强制审批断点，审批前执行线程挂起。
- [ ] 原始 shell 仅作为兜底能力，默认优先结构化只读工具和受控变更工具。
- [ ] 危险动作支持 dry-run、影响范围说明、预期变更摘要。

验收标准：

- [ ] `service.restart`、`kubectl rollout restart`、`config.apply` 等动作在执行前都能展示审批卡。
- [ ] 同类动作可以在审批卡里看到目标主机/服务/环境，而不是只看到一行命令字符串。

### P0-6 异步任务、后台执行与硬中断

- [ ] 长时任务支持后台执行，不依赖单次 HTTP request-response 生命周期。
- [ ] 前端可看到任务状态：排队、执行中、等待审批、已取消、已完成、失败。
- [ ] 用户点击“停止”后，Cancellation Context 必须向下传播到 Agent Loop、Host Agent、外部 API 请求。
- [ ] 停止失败时也要明确返回“哪些步骤已停、哪些仍需人工处理”。

验收标准：

- [ ] 数据备份、批量回滚、日志归档等长任务可以持续推送进度且不中断聊天线程。
- [ ] 点击停止后，界面能在合理时间内反馈真实停止结果，而不是只本地改状态。

### P0-7 执行后自动验证与回滚闭环

- [ ] 每个变更动作都要定义最小验证集：成功条件、观察窗口、失败判定。
- [ ] 动作完成后自动触发验证，不允许“执行成功”直接等价于“问题已解决”。
- [ ] 验证失败时，优先给出回滚或下一步隔离建议。
- [ ] 审批卡和结果卡里都要能看到验证状态。

验收标准：

- [ ] 重启服务后，系统自动检查错误率/延迟/健康探针，而不是只返回“已执行完成”。
- [ ] 若恢复失败，线程里会继续给出“回滚 / 扩容 / 查看最近变更”等下一步。

### P0-8 审计、时间线与可回放

- [ ] 统一沉淀会话级 incident timeline：用户输入、计划、证据获取、审批、执行、验证、停止、失败。
- [ ] 任一步骤支持事后回放和筛选。
- [ ] 审计数据至少覆盖：人、时间、目标、动作、参数、证据、决策、结果。

验收标准：

- [ ] 事故结束后，可直接导出简化版处置记录和复盘摘要。
- [ ] 审批拒绝、执行失败、人工停止都会在时间线上有独立事件。

### P0-9 结构化卡片与结果容器

- [ ] 至少提供并统一接入以下卡片：指标卡、日志卡、Trace 卡、审批卡、变更卡、验证卡、回滚卡、总结卡。
- [ ] 指标查询结果优先渲染为图表或表格，不要求模型把海量数据压成纯文本。
- [ ] 日志/终端输出支持折叠、筛选、复制、定位来源。

验收标准：

- [ ] 用户能在一次 incident 处理流程里同时看到正文、卡片、时间线和证据面板。
- [ ] 无需离开聊天上下文就能查看关键指标图、日志片段和审批详情。

## 6. P1 增强项

### P1-1 动态上下文 Hook

- [ ] 每轮推理前自动注入当前告警、环境健康、最近变更、最近失败审批、当班信息。
- [ ] 支持按故障类型自动挂载内部 SOP / Runbook。

### P1-2 拓扑与影响面感知

- [ ] 基于服务依赖、主机归属、集群、租户和环境信息展示 blast radius。
- [ ] 当动作可能跨环境或跨租户时，强制二次确认。

### P1-3 不确定性表达

- [ ] 输出候选根因列表、当前置信度和下一步最小取证动作。
- [ ] 对“证据不足”的场景禁止过度自信表达。

### P1-4 结构化下一步动作

- [ ] 每个关键结论后生成建议动作按钮，如“按实例展开”“对比发布前后”“查看最近配置变更”“生成回滚方案”。

### P1-5 Incident 协同与交接

- [ ] 支持交接摘要、当前假设、已执行动作、未完成待办。
- [ ] 支持多人共享同一 incident 的时间线和审批结果。

### P1-6 外部系统联动

- [ ] 与 Slack / PagerDuty / Jira / 飞书等系统做最小同步，避免多处重复录入。

## 7. P2 后续项

- [ ] 自动从成功处置中提炼可复用 SOP 草稿，但默认不自动上线。
- [ ] 支持更细粒度的风险策略引擎，如按环境、服务等级、时间窗口差异化审批。
- [ ] 支持 Incident 结束后一键生成复盘报告模板。

## 8. 非目标与边界

以下能力本阶段不作为目标，避免需求膨胀：

- [ ] 不追求展示完整思维链原文，只要求阶段、动作、证据和决策足够透明。
- [ ] 不允许“无人审批自动修改生产”成为默认路径。
- [ ] 不把自由生成 shell 作为主要交互范式，原始命令能力仅作兜底。
- [ ] 不为了追求聊天自然度而牺牲审计、权限和执行可控性。

## 9. 面向当前项目的落地映射

建议按现有模块拆解，而不是另起炉灶：

- [ ] `web/`：补齐模式切换、timeline、evidence、approval、verification、rollback 等 UI 容器。
- [ ] `internal/agentloop/`：补齐阶段事件、执行状态机、取消传播、证据引用模型。
- [ ] `internal/server/`：补齐计划审批、执行审批、验证回流、时间线聚合。
- [ ] `internal/model/`：增加 `mode`、`stage`、`risk_level`、`evidence_ref`、`verification_status`、`rollback_hint` 等字段。
- [ ] `internal/orchestrator/`：面向 incident 而不是单任务调度，支持计划 -> 执行 -> 验证闭环。
- [ ] `cmd/host-agent/` / `internal/agentrpc/`：补齐 cancellation、dry-run、身份透传、结果结构化返回。
- [ ] `Coroot / MCP / UI Card`：继续作为结构化证据与卡片承载层，不再让模型单独“口述图表”。

## 10. 总体验收口径

以下 6 个场景全部打通后，才能认为本轮改造达标：

- [ ] 场景 1：用户发起“订单服务延迟升高，帮我排查”，系统能展示计划、证据、假设排序与结构化下一步。
- [ ] 场景 2：用户同意执行“重启异常实例”，系统能先审批、再执行、再自动验证恢复情况。
- [ ] 场景 3：执行过程中用户点击停止，底层任务被真实中断，且界面反馈剩余风险。
- [ ] 场景 4：系统基于 OOMKilled 自动挂载 SOP，指导排查而不是只给泛化建议。
- [ ] 场景 5：事故结束后可导出处置时间线、审批记录、证据摘要和验证结果。
- [ ] 场景 6：权限较低用户无法借助 Agent 绕过 RBAC 执行超权限动作。

## 11. 一句话结论

这轮改造的目标不是把 Chat 做得“更像专家聊天”，而是把它做成一个**证据驱动、计划前置、权限受控、可验证、可回放的运维 Incident 操作台**。
