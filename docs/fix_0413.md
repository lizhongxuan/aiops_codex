# Bifrost Runtime 排查与修复文档 (2026-04-13)

## 一、问题现象

使用 Bifrost 方案替换 Codex Runtime 后，同样使用 gpt-5.4，AI 表现明显变差：
1. 不主动调用工具（web_search、execute_readonly_query）
2. 搜索质量差，经常说"不能联网"
3. 命令执行失败或不触发
4. 整体回答质量下降，感觉"弱智了"

## 二、根因分析

### 根因 1：reasoning_content 被丢弃（最关键）

gpt-5.4 通过 Chat Completions API 返回的流式响应包含 `reasoning_content` 字段（思维链/Chain-of-Thought），但 Bifrost 的 OpenAI provider 只处理 `content` 字段：

```json
// 模型实际返回的 delta:
{"delta": {"role": "assistant", "reasoning_content": "I should use web_search..."}}

// 我们的解析器只看 content:
if choice.Delta.Content != "" {  // reasoning_content 被忽略！
    ch <- StreamEvent{Type: "content_delta", Delta: choice.Delta.Content}
}
```

影响：模型的推理过程被截断，导致工具调用决策不完整。

**修复方案**：在 `openAIStreamDelta` 结构体中添加 `ReasoningContent` 字段，在流式解析中同时处理。

### 根因 2：代理路由不稳定

你的 LLM 代理 (`127.0.0.1:8317`) 有时把请求转发到 `chatgpt.com/backend-api/codex/responses`（Codex Responses API），而不是标准的 Chat Completions 端点。这导致间歇性 500 EOF 错误。

### 根因 3：Responses API 与 Chat Completions API 混用

之前尝试把所有请求切换到 Responses API，但代理对 Responses API 的支持不稳定，导致所有请求都失败。

## 三、工具（Tools）实现方式说明

### 工具分两类：

#### A. 我们自己实现的工具（Function Tools）

这些工具通过 OpenAI Chat Completions API 的 `tools` 参数发送给模型，模型决定是否调用，调用后由我们的代码执行：

| 工具名 | 实现位置 | 执行方式 | 说明 |
|--------|---------|---------|------|
| `execute_readonly_query` | `bifrost_runtime.go` | `os/exec` 本地执行 | 只读命令 |
| `execute_command` | `bifrost_runtime.go` | `os/exec` 本地执行 | 需审批的变更命令 |
| `write_file` | `bifrost_runtime.go` | `os.WriteFile` | 文件写入 |
| `web_search` | `tools_websearch.go` | DuckDuckGo HTML 抓取 | 搜索质量一般 |
| `open_page` | `tools_websearch.go` | HTTP GET | 抓取网页内容 |
| `find_in_page` | `tools_websearch.go` | 字符串搜索 | 页内搜索 |
| `ask_user_question` | `bifrost_runtime.go` | WebSocket 推送 | 弹出选择卡片 |
| `list_files` | `bifrost_runtime.go` | gRPC 远程执行 | 远程文件列表 |
| `read_file` | `bifrost_runtime.go` | gRPC 远程执行 | 远程文件读取 |
| `search_files` | `bifrost_runtime.go` | gRPC 远程执行 | 远程文件搜索 |

这些工具对所有 LLM 都通用（GPT/Claude/DeepSeek/GLM/Ollama），因为它们使用标准的 OpenAI Function Calling 格式。

#### B. LLM Provider 内置工具（Native Tools）

这些工具由 LLM 服务商在服务端实现，客户端只需声明启用：

| Provider | 内置工具 | API 格式 | 说明 |
|----------|---------|---------|------|
| OpenAI | `web_search` | `{"type": "web_search"}` (Responses API) | Bing 搜索，质量最高 |
| OpenAI | `code_interpreter` | `{"type": "code_interpreter"}` | 服务端代码执行 |
| Anthropic | `web_search` | Claude 内置 | Anthropic 自有搜索 |
| DeepSeek | 无内置搜索 | - | 只支持 Function Calling |
| GLM | `web_search` | GLM 自有格式 | 智谱自有搜索 |

**关键区别**：内置工具的搜索在 LLM 服务端执行（用 Bing/Google），质量远高于我们自己的 DuckDuckGo 抓取。

## 四、推荐的工具配置架构

### 核心原则：在配置 LLM 时就定义好用什么工具，不靠 fallback

```
┌─────────────────────────────────────────────────────────┐
│ LLM Provider 配置                                        │
│                                                          │
│ provider: openai                                         │
│ model: gpt-5.4                                           │
│ web_search_mode: native    ← 用 OpenAI 内置搜索          │
│ api_mode: responses        ← 用 Responses API            │
│                                                          │
│ provider: anthropic                                      │
│ model: claude-sonnet-4                                   │
│ web_search_mode: native    ← 用 Anthropic 内置搜索       │
│                                                          │
│ provider: deepseek                                       │
│ model: deepseek-chat                                     │
│ web_search_mode: brave     ← 用 Brave Search API         │
│                                                          │
│ provider: ollama                                         │
│ model: qwen3:72b                                         │
│ web_search_mode: duckduckgo ← 用 DuckDuckGo 抓取         │
└─────────────────────────────────────────────────────────┘
```

### 建议的环境变量配置

```bash
# 搜索工具配置
WEB_SEARCH_MODE=native|brave|duckduckgo|disabled
# native: 使用 LLM provider 内置搜索（OpenAI/Anthropic/GLM）
# brave: 使用 Brave Search API（需要 BRAVE_API_KEY）
# duckduckgo: 使用 DuckDuckGo HTML 抓取（免费但质量低）
# disabled: 禁用搜索

BRAVE_API_KEY=           # Brave Search API key（可选）

# API 模式配置
LLM_API_MODE=chat|responses
# chat: 标准 Chat Completions API（默认，兼容所有 provider）
# responses: OpenAI Responses API（仅 OpenAI，支持内置 web_search）
```
